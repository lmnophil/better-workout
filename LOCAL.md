# Local Development

Two ways to run the app on your laptop:

- **`npm` mode** — Postgres in Docker, app via `next dev`. Hot reload, fast iteration. Day-to-day default.
- **Compose mode** — full stack (db + app + backup) under Docker Compose, identical to production. Use this when you want to verify the production boot path before shipping.

This doc covers setup, seeding, and how to reset either environment to a known-empty state. The reset paths matter: they're the same operations that will run when you spin up production for the first time, so rehearsing them locally is the closest thing to a dry run you've got.

For the actual production deployment guide (Proxmox / Oracle Cloud / DNS / TLS / backups), see [`DEPLOY.md`](./DEPLOY.md).

## Prerequisites

- **Node 20.19+** (matches `package.json` `engines.node`)
- **Docker** (for Postgres in either mode)
- **npm** (no yarn/pnpm support configured)

WSL2 / macOS / Linux all work — nothing OS-specific in this stack.

## Path A: npm mode (everyday work)

### First-time setup

```bash
# 1. Install dependencies
npm install

# 2. Start a local Postgres
docker run -d --name workout-pg \
  -e POSTGRES_PASSWORD=devpass \
  -e POSTGRES_DB=workout \
  -p 5432:5432 \
  postgres:16

# 3. Configure environment
cp .env.example .env
./scripts/generate-secrets.sh >> .env   # appends AUTH_SECRET, POSTGRES_PASSWORD, METRICS_TOKEN
$EDITOR .env                             # OAuth/Resend can stay blank if you only want to verify boot
```

`.env` is read by both modes. The vars npm mode actually reads:

- `DATABASE_URL` — defaults to `postgres://postgres:devpass@localhost:5432/workout` (matches the docker run above)
- `AUTH_SECRET` — generated by the script
- `AUTH_URL=http://localhost:3000`
- OAuth / Resend creds — optional. Without them the relevant sign-in path is unavailable but the app still boots (warnings only — see [`lib/env.ts`](lib/env.ts))

```bash
# 4. Apply migrations and seed
npm run db:migrate     # `prisma migrate dev` — creates schema from prisma/migrations
npm run db:seed        # built-in exercises + starter templates

# 5. Run the app
npm run dev
# or with pretty logs:
npm run dev:pretty
```

Visit `http://localhost:3000` — you'll be redirected to `/signin`.

### Subsequent runs

```bash
docker start workout-pg
npm run dev
```

If `schema.prisma` changed since you last pulled:

```bash
npm run db:migrate     # picks up new migrations
```

### Useful commands in npm mode

```bash
npm run typecheck      # tsc --noEmit — run after meaningful changes
npm run db:studio      # browser GUI for the DB
npm run db:seed        # idempotent — re-running won't duplicate; refreshes built-ins from lib/exercises-data.ts
```

## Path B: Compose mode (mirrors production)

Use this when you want to confirm the production stack — env validation, entrypoint, migrate-on-boot, healthcheck, backup loop — actually works before shipping. Slower to iterate (every code change needs a rebuild) but it's what prod will do.

### First-time setup

```bash
# 1. Configure environment (same .env as npm mode)
cp .env.example .env
./scripts/generate-secrets.sh >> .env
$EDITOR .env
```

For local compose you need at minimum:

- `AUTH_SECRET`, `AUTH_URL=http://localhost:3000`, OAuth/Resend (or leave blank — see [`lib/env.ts`](lib/env.ts) for which are hard-required vs warned)
- `POSTGRES_PASSWORD` (generated)
- `BACKUP_HOST_DIR=./backups` — the repo has an empty `backups/` directory ready. Compose hard-requires this var (`${BACKUP_HOST_DIR:?...}` will refuse to start without it)

`DATABASE_URL` is constructed by compose itself from the `POSTGRES_*` vars; you don't set it for this mode.

```bash
# 2. Build and start
docker compose up -d --build

# 3. Watch the entrypoint apply migrations and start the server
docker compose logs -f app
# Look for "→ Applying database migrations..." then "→ Starting Next.js server..."

# 4. Seed
docker compose exec app node prisma/seed.js
```

The seed script is bundled to `prisma/seed.js` at build time — that's why you use `node` (not `tsx`) inside the container, and why you use `exec` not `run` (the entrypoint hardcodes `node server.js` and ignores `run` args). Both are documented in [`prisma/CLAUDE.md`](prisma/CLAUDE.md).

Visit `http://localhost:3000`.

### Useful commands in compose mode

```bash
docker compose ps                                        # see what's up
docker compose logs -f app                               # tail app logs
docker compose exec db psql -U workout -d workout        # DB shell
docker compose exec backup sh /usr/local/bin/backup.sh   # manual backup
```

## Seeding: how it works

Both modes call the same seed (`prisma/seed.ts`) but invoke it differently:

| Mode    | Command                                           | What runs                                     |
|---------|---------------------------------------------------|-----------------------------------------------|
| npm     | `npm run db:seed`                                 | `tsx prisma/seed.ts` (devDep `tsx` required)  |
| compose | `docker compose exec app node prisma/seed.js`     | Bundled output from the Docker build (no tsx) |

Both are **idempotent**. The seed upserts by `(ownerId=null, name)` for built-in exercises and by `(userId=null, isBuiltin=true, name)` for starter templates. Re-running after edits to `lib/exercises-data.ts` updates built-ins in place; re-running with no edits is a no-op.

If you only changed the seed (not the schema), you don't need to reset the DB — just re-seed.

## Reset to a fresh state

This is the path that matters for "what will my first prod deploy look like." Pick the recipe that matches your mode.

### npm mode: nuclear option (recommended)

```bash
npx prisma migrate reset
```

Drops the database, re-applies every migration from scratch, runs the seed (configured via `migrations.seed` in [`prisma.config.ts`](prisma.config.ts)). Confirms before destroying.

This is the closest single-command equivalent to what a brand-new prod database goes through, minus the entrypoint wrapper.

### npm mode: nuke the Postgres volume entirely

If you want to be sure nothing sticks around (extensions, roles, anything outside `public`):

```bash
docker rm -fv workout-pg                              # stops and deletes container + volume
docker run -d --name workout-pg \
  -e POSTGRES_PASSWORD=devpass \
  -e POSTGRES_DB=workout \
  -p 5432:5432 \
  postgres:16

npm run db:migrate
npm run db:seed
```

### Compose mode: nuclear option (recommended)

```bash
docker compose down -v
```

The `-v` is critical — without it the `postgres-data` named volume sticks around and you get the old DB back when you `up` again. `down -v` removes it.

Then bring it back up:

```bash
docker compose up -d --build
docker compose logs -f app                            # entrypoint runs `prisma migrate deploy`
docker compose exec app node prisma/seed.js
```

This is the most faithful rehearsal of a first prod deploy — same image build, same entrypoint, same migrate path, same seed invocation.

### Compose mode: clear data without rebuilding

If you want to reset the data but keep the running containers (faster):

```bash
docker compose exec db psql -U workout -d workout -v ON_ERROR_STOP=1 <<'SQL'
DROP SCHEMA IF EXISTS public CASCADE;
CREATE SCHEMA public;
GRANT ALL ON SCHEMA public TO public;
SQL

docker compose restart app                            # entrypoint re-runs migrate deploy
docker compose exec app node prisma/seed.js
```

Same drop-schema dance [`scripts/restore.sh`](scripts/restore.sh) uses before piping in a backup. Avoid `prisma migrate reset` from inside the runtime container — it would try to invoke the seed via `tsx`, which isn't in the prod image.

## Verifying a clean first deploy

Before pointing prod at a real domain for the first time, run through this in compose mode locally — it's the closest dry run you can do:

```bash
# 1. Start from absolute scratch
docker compose down -v

# 2. Rebuild and boot (same path prod takes on first start)
docker compose up -d --build

# 3. Confirm env validation passed
docker compose logs app | grep -E "env\.(error|warning|validated)"
# Want: an "env.validated" line. No "env.error" lines.
# "env.warning" is fine if you intentionally left OAuth/Resend blank.

# 4. Confirm migrations applied
docker compose logs app | grep -i "migrat"
# Want: "Applying database migrations..." with no errors after.

# 5. Confirm the healthcheck passes
docker compose ps
# Want: app status "healthy" (takes ~30s after boot)

curl -s http://localhost:3000/api/healthz
# Want: {"status":"ok","durationMs":...}

# 6. Seed and confirm built-ins land
docker compose exec app node prisma/seed.js
# Want on a fresh DB: "Exercises — created: N, updated: 0"

# 7. Open the app, sign in (if you wired OAuth/Resend), confirm the exercise
#    picker shows the built-ins and you can start a session.
```

If any step fails locally, it'll fail on prod too — fix it now. If they all pass, the same `docker compose up -d --build` on the prod box will behave the same way.

## Switching between modes

The two modes share `.env` but use different Postgres instances:

- npm mode talks to Postgres on `localhost:5432` (the standalone `workout-pg` container)
- Compose mode talks to Postgres inside the compose network (`db:5432`)

Their data is independent. Switching modes doesn't migrate data between them — by design, since compose mode is meant to be a clean rehearsal. To move data between, take a `pg_dump` from one and pipe it into the other.

If you do want npm mode to use the compose Postgres instead of the standalone one, publish the compose db port to the host (add `ports: ['5432:5432']` to the `db` service) and update `DATABASE_URL` in `.env` to point at it. Not recommended unless you have a reason — they're easier to reason about kept separate.

## Troubleshooting

**`Environment validation failed. See errors above.`** — [`lib/env.ts`](lib/env.ts) rejected a missing/malformed required var. The fatal log line just above says which one. Common culprits: blank `AUTH_SECRET`, `AUTH_URL` without a protocol (`localhost:3000` instead of `http://localhost:3000`), `DATABASE_URL` with the wrong scheme.

**`BACKUP_HOST_DIR must be set in .env`** — compose-only. Set it to `./backups` for local compose (the repo has an empty `backups/` directory ready). Required even locally because [`docker-compose.yml`](docker-compose.yml) enforces it on the volume mount.

**`Can't reach database server at db:5432` (compose)** — db container hasn't passed its healthcheck yet, or you `up`'d only the app. `docker compose up -d` brings everything up; the app waits on db's healthcheck via `depends_on.condition`.

**`Can't reach database server at localhost:5432` (npm)** — the `workout-pg` container isn't running. `docker start workout-pg`.

**Seed says "0 created, 0 updated"** — DB already has the built-ins. Not an error; the seed is idempotent.

**`prisma migrate reset` errors with "permission denied to drop database"** — your `DATABASE_URL` user isn't a superuser. The default `postgres` user (npm mode) and the `workout` user the postgres image creates from `POSTGRES_USER` (compose mode) both are; if you've overridden things, check.

**Service worker doesn't pick up code changes** — service workers cache aggressively. Hard-refresh (Cmd+Shift+R / Ctrl+F5) or unregister via DevTools → Application → Service Workers.

## What's different between local and a real prod deploy

In case you're wondering what your local rehearsal *doesn't* catch:

- **Reverse proxy / TLS** — local has none; prod has Caddy/nginx/Traefik in front (see [`docs/caddy-snippet.example`](docs/caddy-snippet.example)).
- **OAuth callback URL** — local uses `http://localhost:3000/api/auth/callback/google`; prod uses your real domain. Both must be registered in Google Cloud Console for the matching `AUTH_URL`.
- **Magic links via Resend** — locally with `AUTH_EMAIL_FROM=onboarding@resend.dev`, only mail to your Resend signup address arrives. Use Google sign-in for everyone-else dev.
- **Backups** — the backup container runs locally too (writes to `./backups`), but you don't have an offsite pipeline picking files up. That's fine for local; not fine for prod (see [`DEPLOY.md`](DEPLOY.md#backups)).
- **`NODE_ENV`** — npm mode runs as `development`; compose mode sets `NODE_ENV=production` (per [`docker-compose.yml`](docker-compose.yml)), which engages the prod-required env checks (e.g. `METRICS_TOKEN`).
